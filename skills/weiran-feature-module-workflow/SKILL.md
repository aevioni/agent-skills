# 功能模块全流程开发工作流

每个功能模块的开发遵循严格的文档驱动流程：先从产品文档提取需求，生成技术需求文档并经人工确认，再生成技术方案设计并经人工确认，然后基于方案实现代码，开发完成后并发生成前端对接文档、单元测试文档和前端汇总文档，最后自动执行单元测试与代码审查。文档和代码是一体的，任何代码改动必须同步更新所有引用该改动的文档。

## Why This Matters

不遵循此流程会导致：文档与代码不一致、前端对接文档缺失、数据库变更漏提交、总进度失联、前端接口汇总遗漏。每个遗漏都会造成联调时的返工。人工确认门控确保技术需求和方案设计在编码前就已对齐，避免开发返工。

## Recognition Pattern

当用户的指令涉及功能模块的文档整理或开发时触发此工作流。关键信号：

**直接触发：**
- 提到技术文档目录路径
- 提到产品文档
- 要求创建某功能模块的技术文档
- 要求根据文档进行开发

**间接触发（需满足复杂度阈值）：**
- "新增 XX 功能"、"开发 XX 模块"、"实现 XX 需求"
- "把 XX 功能做完"、"继续开发 XX"

**复杂度阈值**（满足任意一条才走全流程，否则按常规开发）：
- 涉及 **2 个以上文件** 的功能性改动（非 bugfix、非单点优化）
- 需要新增数据库表或字段
- 需要新增 API 路由
- 涉及跨模块联动

> 单文件 bugfix、单点优化、纯样式调整等不触发全流程，按常规开发即可。

## 项目适配（使用前必须配置）

在应用本工作流之前，需要根据项目实际情况配置以下变量。可在项目的 `CLAUDE.md` 或技能配置中定义：

| 变量 | 说明 | 示例 |
|---|---|---|
| `{DOC_ROOT}` | 技术文档根目录 | `docs/tech/`、`README/feat1.7/[TECH]技术文档/` |
| `{PRODUCT_DOC}` | 产品文档路径 | `docs/product/`、`README/feat1.7/[PM]产品文档/` |
| `{SQL_PATH}` | SQL 变更文件路径 | `migrations/`、`resources/sql/v1.7.sql` |
| `{TEST_DIR}` | 测试目录 | `tests/`、`modules/{module}/tests/` |
| `{TEST_CMD}` | 测试运行命令 | `phpunit`、`npm test`、`pytest` |
| `{FRONT_API_SUMMARY}` | 前端接口汇总文档路径 | `docs/api-summary.md` |
| `{PROGRESS_DOC}` | 总进度文档路径 | `docs/progress.md` |

## The Approach

### 流程总览

```
阶段一：需求调研 → 阶段二：技术需求文档 → ⏸️ 人工确认
→ 阶段三：技术方案设计 → ⏸️ 人工确认
→ 阶段四：代码开发（含数据库变更同步）
→ 阶段五：文档并发生成（FrontApi + TechTest + 前端汇总 + 文档状态同步）
→ 阶段六：自动执行单元测试
→ 阶段七：代码审查
```

### 多 Agent 并行策略

本工作流**默认并行执行**，除非明确不满足并行条件才降级为串行。

**每个阶段开始前，执行者必须先做「并行决策」，不允许跳过。**

#### 并行决策流程（每个阶段入口必须执行）

```
1. 查并行时机总览表 → 当前阶段标注了哪种并行方式？
2. 检查排除条件 → 改动项 < 3 个？模块间有强依赖？用户要求串行？
   - 是 → 串行执行，在响应中注明"本阶段串行，原因：xxx"
   - 否 → 进入步骤 3
3. 拆分任务 → 按总览表拆为 N 个独立子任务
4. 在同一条消息中发出所有 Agent 调用 → 这是并行的唯一实现方式
5. 等待全部返回 → 汇合检查一致性
```

**不执行并行决策就直接动手 = 违反工作流规范。**

### 并行时机总览

| 阶段 | 并行方式 | 说明 |
|---|---|---|
| 阶段一 调研 | `explore` x 2-3 并行 | 读文档 + 查代码 + 验证已完成项可同时进行 |
| 阶段二 技术需求 | 不并行 | 单文档生成，完成后等待人工确认 |
| 阶段三 技术方案 | 不并行 | 单文档生成，完成后等待人工确认 |
| 阶段四 开发 | `executor` x N 并行（按 Action 拆分） | 不同 Action 之间无依赖时可并行开发+测试，**同一文件禁止并行写入** |
| 阶段五 文档生成 | `executor` x 3-4 并行 | FrontApi + TechTest + 前端接口汇总 + 文档状态同步可并行 |
| 阶段六 测试 | `test-engineer` x 1 | 自动执行全量单元测试 |
| 阶段七 审查 | `code-reviewer` x 1 | 独立 agent 审查，与开发 agent 分离 |

### 并行执行机制

**关键原理**：Claude Code 中实现并行的唯一方式是**在同一条消息中发出多个 Agent 工具调用**。多个 Agent 调用写在同一个响应中 = 并行执行；分开写在不同响应中 = 串行执行。

```
# 并行：同一条消息中发出多个 Agent 调用，它们会同时启动
Agent(subagent_type="oh-my-claudecode:explore", prompt="任务A")    <- 同时启动
Agent(subagent_type="oh-my-claudecode:explore", prompt="任务B")    <- 同时启动

# 串行：每个 Agent 调用在不同的响应中，前一个完成后才启动下一个
Agent(subagent_type="oh-my-claudecode:explore", prompt="任务A")    <- 完成后
Agent(subagent_type="oh-my-claudecode:explore", prompt="任务B")    <- 才启动这个
```

**`run_in_background: true`**：不需要立即等待结果时，可以让 agent 在后台运行。适合阶段五文档生成等不阻塞后续工作的任务。

### 并行安全规则

1. **文件锁定**：同一文件不允许两个 agent 同时写入。如需修改同一文件，必须串行或拆分为不同代码段
2. **依赖屏障**：阶段间是串行屏障（含人工确认门控），阶段内可并行
3. **汇合检查**：并行 agent 全部完成后，主 agent 必须检查产出物之间的一致性（如多个功能单元的错误码是否冲突、接口定义是否兼容）
4. **失败回滚**：任一并行 agent 失败，暂停其他 agent，修复后重新评估并行计划

## 阶段详解

### 阶段一：需求调研（先看再做） `必须并行`

> **并行指令**：**必须**将以下三组任务拆为 2-3 个 `explore` agent，在同一条消息中发出，并行执行。
> - Agent 1：步骤 1-4（读产品文档+总功能点）
> - Agent 2：步骤 5-6（查代码+验证已完成项）
> - Agent 3：步骤 7-8（排查级联影响+识别复用模式）
> - 等待所有结果返回后，汇合到步骤 9。
>
> **已完成项验证的并行拆分规则**：当文档中标记为已完成的改动项 >= 3 个时，步骤 5-6 的验证工作**必须按改动项分组拆为多个 `explore` agent 并行执行**，不允许单个 agent 串行验证所有改动项。每个 agent 需逐项读取实际代码，给出结论，不能只看文件是否存在。

1. **确定版本/分支**：从当前 git 分支名提取版本号，用于定位文档目录
2. **查重**：检查文档目录下是否已有该模块目录，同时检查是否存在其他人创建的相关文档，避免重复
3. **读产品文档**：在 `{PRODUCT_DOC}` 下找到对应章节
4. **读总功能点**：在技术文档目录下找到技术侧拆解
5. **查现有代码**：grep 相关 Controller、Action、Model、Route 确认实现状态
6. **验证"已完成"项**：对于文档标记为已完成的改动项，**必须逐项验证代码是否真的符合产品要求**，不能只看标记
7. **排查级联影响**：数据变更操作（如修改关联 ID、删除记录、状态变更）需要检查是否有关联数据需要同步处理
8. **识别可复用模式**：在现有代码中找到与当前模块最相关的 Controller/Action/Model 风格，作为开发参照
9. **对齐理解**（汇合点）：收集所有并行 agent 的结果，如果需求存在歧义或与现有代码冲突，先向用户确认再继续

### 阶段二：技术需求文档 `人工确认门控`

在 `{DOC_ROOT}/{模块名}/` 下创建：

| 文件 | 命名规范 | 内容 |
|---|---|---|
| 技术需求 | `[TechDemand]{模块名}技术需求.md` | 目标、产品来源、技术栈约束、需求拆解、数据模型、改动项清单（含状态） |

#### TechDemand 规范要点

- **技术栈约束**：如涉及跨项目协作（如前后端、微服务等），必须列出编号约束清单，明确各项目职责边界、通信方式、鉴权方案等技术决策
- 改动项清单用编号标识（如 D1、D2），便于后续追踪

**生成完成后，必须向用户展示文档内容并等待确认。用户确认无误后才能进入阶段三。**

### 阶段三：技术方案设计 `人工确认门控`

基于已确认的 TechDemand，在同一目录下创建：

| 文件 | 命名规范 | 内容 |
|---|---|---|
| 技术方案 | `[TechDesign]{模块名}技术方案设计.md` | 需求变化分析、现有代码复用点、数据库变更 SQL、实现代码示例、文件改动清单（含状态） |

#### TechDesign 规范要点

- **需求变化分析**：列出本轮需求相比上轮的核心变化点及其对表结构、规则、接口的影响
- **现有代码复用点**：明确列出本模块应参照的既有代码文件路径和复用原因
- 文件改动清单标注状态

**生成完成后，必须向用户展示文档内容并等待确认。用户确认无误后才能进入阶段四。**

### 阶段四：代码开发（按文档改动清单逐项，边开发边测试） `必须并行`

> **并行指令**：
> 1. 基础层（Model+验证规则）是前置依赖，**串行完成**（串行屏障）
> 2. 之后**必须**检查不同 Action 之间是否有依赖。无依赖时，**必须**拆为多个 `executor` agent（每个负责一个 Action + 对应测试），在同一条消息中发出
> 3. 等待全部返回 -> 汇合检查一致性
> 4. Controller/Route/OpenAPI Schema 等涉及共享文件的步骤，由主 agent 串行完成

#### Action 命名规范

- **后台功能**：使用 `ActBeXxx` 命名（Be = Backend），如 `ActBeDeviceList`、`ActBeUserCreate`
- **前台/其他功能**：使用 `ActXxx` 命名，如 `ActLogin`、`ActRegister`

典型改动顺序（按需裁剪，跳过不涉及的步骤）：

1. **Model + Migration**（新表/新字段） — 无新表/新字段时跳过 `串行屏障`
2. **验证规则**（Request scene + rules） — 无新入参时跳过 `串行屏障`
3. **Action 层**（业务逻辑方法） -> **立即编写对应单元测试并运行** `可按 Action 拆分并行`
4. **Controller + OpenAPI Attributes**（路由 + 权限 + 方法签名）
5. **Route 注册** — 无新路由时跳过
6. **OpenAPI Schema**（响应体定义） — 响应结构未变时跳过
7. **权限注册**（Controller 构造函数 `self::$permission` + `permissions.yaml`）— 新增接口时必须检查权限配置是否完整
8. **错误信息中文化** — Action 层 `setError()` 错误信息必须中文，Request 类 `attributes()` 必须为所有字段定义中文名

> **边开发边测试原则**：每完成一个功能单元后，立即编写该单元的单元测试并运行通过，再进入下一个。不要等全部开发完再集中写测试。

#### 数据库变更同步（阶段四附属步骤）

代码开发完成后，将建表/ALTER 语句追加到 `{SQL_PATH}`：
- 用注释块分隔各模块
- CREATE TABLE 用 `IF NOT EXISTS`
- **必须有详细备注**：每个字段必须有注释说明，表必须有注释说明
- 无数据库变更时跳过

### 阶段五：文档并发生成 `必须并行`

> **并行指令**：以下文档互不依赖，**必须**拆为多个 `executor` agent 在同一条消息中发出并行生成：
> - Agent 1：生成 FrontApi 前端对接文档
> - Agent 2：生成 TechTest 单元测试文档框架
> - Agent 3：更新前端接口汇总（功能章节+权限汇总+接口总览）
> - Agent 4（`run_in_background=true`）：更新本模块 TechDemand+TechDesign 状态 + 总进度

在 `{DOC_ROOT}/{模块名}/` 下创建：

| 文件 | 命名规范 | 内容 |
|---|---|---|
| 前端对接 | `[FrontApi]{模块名}前端对接文档.md` | 接口变更、请求/响应字段、前端改动项 |
| 测试文档 | `[TechTest]{模块名}技术单元测试文档.md` | 测试策略、预期测试文件清单、覆盖维度规划 |

#### FrontApi 规范要点

- 只列变更（【新增】【修改】标记），不列未改动的 API
- 区分后端改动项 vs 前端改动项
- 破坏性变更用醒目标注 + 变更前/变更后对比
- 改动项状态用 `待完成` / `已完成` / `待前端`
- **联调测试**：如果模块提供了测试页面或联调工具，在文档最前面说明测试地址和使用方式
- **前端推荐对接流程**：对于接口较多的模块，提供页面初始化顺序和用户操作后的联动规则

#### TechTest 规范要点

**阶段五 — 写测试框架：**
- **测试策略**：说明本模块的测试分层（纯单元 / 集成 / HTTP）和选择理由
- **预期测试文件清单**：表格列出计划创建的文件路径、测试类名、覆盖目标，状态标 `待完成`
- **覆盖维度规划**：从业务维度规划需要交叉覆盖的矩阵

**阶段六测试后 — 补充实际结果：**
- **测试文件清单**：更新实际测试数/断言数、状态标 `已完成`
- **用例详情**：按测试类分组，用表格列出每个测试方法的验证点和预期结果
- **覆盖矩阵**：用实际数据填充，标注已覆盖/未覆盖
- **运行命令**：提供全量运行、单文件运行、单方法运行的 CLI 示例
- **待补充建议**：列出未覆盖的场景、建议测试层级、优先级

#### 文档状态同步

完成代码后必须同步更新以下文档：

1. **本模块文档**：TechDemand + TechDesign 中改动项状态 -> `已完成`
2. **总进度**：`{PROGRESS_DOC}` — 新增行或更新状态
3. **前端接口汇总**：`{FRONT_API_SUMMARY}`
   - 功能章节（新功能往后追加）
   - **权限汇总表** — 每个新增接口都要在权限汇总中列出对应的权限标识、说明、接口路径、变更标记
   - 接口总览表（更新合计数）

> **易遗漏项**：权限汇总是最容易漏掉的。每次新增接口后必须检查权限汇总表是否包含该接口。

### 阶段六：自动执行单元测试

> 阶段四已完成边开发边测试，此阶段做全量回归并用实际结果回填 TechTest 文档。

- **全量回归**：运行本模块所有测试（`{TEST_CMD} {TEST_DIR}`），确保边开发边测试阶段的用例仍然全部通过
- 补充阶段四未覆盖的边界用例和集成测试
- 用实际测试结果回填 TechTest 文档（测试文件清单、用例详情、覆盖矩阵、运行命令）

### 阶段七：代码审查（独立审查，不自批自审）

**必须委托 `code-reviewer` agent 进行独立审查**，不允许开发者自己审自己。这遵循 authoring/review 分离原则 — 编写代码和审查代码必须是不同的 agent pass。

审查清单（交给 `code-reviewer` 的检查要点）：
- 破坏性变更是否有向后兼容处理或明确标注
- OpenAPI Attributes 与实际返回结构是否一致
- **权限三处一致**：Controller `self::$permission` 映射 <-> `permissions.yaml` slug <-> 前端接口汇总权限表
- 数据库变更语句是否幂等（IF NOT EXISTS / IF EXISTS）
- 新增接口是否已计入前端接口汇总的接口总览合计数
- **级联影响**：数据变更操作是否遗漏了关联数据的同步处理
- 错误信息是否全部为中文
- 边开发边测试阶段的测试是否全部通过
- **数据库表结构验证**：数据库实际表结构与 TechDesign 文档、SQL 文件、Model 代码三方一致

### 全流程完成通知（强制）

**阶段七审查通过后，必须明确告知用户"全流程已完成"**，汇报内容包括：
- 本次完成的模块名称
- 各阶段完成状态总览（调研 -> 需求 -> 方案 -> 开发 -> 文档 -> 测试 -> 审查）
- 改动文件清单（代码 + SQL + 文档）
- 测试通过情况（测试数/断言数/通过率）
- 审查结论

> 不允许全流程跑完后静默结束。用户必须收到明确的完成通知，才能判断是否可以进入下一个模块。

## 异常处理

| 场景 | 处理方式 |
|---|---|
| 需求理解有误（开发中途发现） | 停止开发 -> 回退未完成代码（`git stash`）-> 回到阶段一重新对齐 -> 更新文档后继续 |
| 测试不通过 | 修复代码 -> 重跑测试 -> 更新文档中受影响的改动项状态，不要标记为已完成直到测试通过 |
| 与现有功能冲突 | 先查 git blame 确认原改动意图 -> 向用户确认优先级 -> 在 TechDemand 中记录冲突决策 |
| 产品文档与总功能点不一致 | 向用户确认以哪个为准 -> 在 TechDemand 中注明差异和最终决策 |
| **人工确认被驳回** | 根据用户反馈修改对应文档 -> 重新提交确认，直到通过 |
| **会话中断（上下文用完/意外断开）** | 见下方「断点恢复」流程 |

### 断点恢复（跨会话接续）

当新会话需要继续上一次未完成的开发时，按以下步骤恢复进度：

1. **读文档状态**：打开本模块的 TechDemand + TechDesign，扫描所有改动项的状态标记
   - `已完成` -> **必须多 agent 并行验证实际代码**（见下方规则），不能只看标记
   - `待完成` -> 从这里继续
   - 无状态标记 -> 视为未开始
2. **读总进度**：`{PROGRESS_DOC}` 确认模块整体处于哪个阶段
3. **查 git diff**：`git diff --name-only` 检查是否有未提交的改动，判断上次中断在哪一步
4. **并行验证已完成项**（强制规则）：当已完成项 >= 3 个时，**必须按改动项分组拆为多个 `explore` agent 并行验证**，每个 agent 逐项读取实际代码确认实现正确性。已完成项 < 3 个时可单 agent 验证
5. **确定恢复点**：根据文档状态 + 代码验证结果 + git diff，定位到具体阶段和改动项，从断点继续
6. **向用户确认**：简要汇报恢复点和验证结果，用户确认后再开始

> 文档中的状态标记就是天然的断点存档。这也是为什么阶段五（文档状态同步）必须及时执行 — 它不仅是给前端看的，也是给下一次会话恢复用的。

## Examples

完整流程实例（复杂功能 — 跨项目、接口多、测试多，充分并行）：

```
1. 调研    -> 3 个 explore 并行：读文档 | 查代码状态 | 验证已完成项
              -> 汇合：技术栈约束 + 复用点确认
2. 技术需求 -> 生成 TechDemand -> 用户确认通过
3. 技术方案 -> 生成 TechDesign -> 用户确认通过
4. 开发    -> Model -> Migration -> 验证规则（串行屏障）
              -> 3 个 executor 并行：Action A+测试 | Action B+测试 | Action C+测试
              -> 汇合 -> Controller -> Route -> OpenAPI Schema（串行）
              -> SQL 追加到 {version}.sql
5. 文档生成 -> 4 个 executor 并行：FrontApi | TechTest | 前端接口汇总 | 文档状态+总进度
6. 测试    -> test-engineer 全量回归 -> 回填 TechTest 实际结果
7. 审查    -> 委托 code-reviewer：接口文档 / 权限 / SQL 幂等 / 测试全通过
```

精简流程实例（仅修改现有接口返回字段，无新表/新路由，不并行）：

```
1. 调研    -> 确认模块文档已存在，查现有代码定位改动点（量少，串行即可）
2. 技术需求 -> TechDemand 新增改动项 -> 用户确认通过
3. 技术方案 -> TechDesign 更新文件改动清单 -> 用户确认通过
4. 开发    -> Action 层修改(->写测试->跑通) -> Controller OpenAPI Attributes 更新 -> OpenAPI Schema 更新
             （跳过 Model/Migration/验证规则/Route/SQL）
5. 文档生成 -> FrontApi 标记【修改】+ TechTest 更新覆盖范围 + 文档状态同步（量少，串行即可）
6. 测试    -> 全量回归通过 -> TechTest 更新实际结果
7. 审查    -> 委托 code-reviewer：OpenAPI Attributes 与返回结构一致
```

断点恢复实例（上次会话中断，D1-D3 已完成，D4 未开始）：

```
1. 恢复    -> 读 TechDemand 状态：D1-D3 已完成，D4 待完成
              -> 读总进度：模块标记"开发中"
              -> git diff：无未提交改动（上次已 commit）
              -> 向用户确认："从 D4 的 Action 层开始继续"
2. 继续开发 -> D4 Action -> 写测试 -> 跑通 -> Controller -> OpenAPI
3. SQL     -> 追加 D4 涉及的 ALTER 语句
4. 文档生成 -> FrontApi 更新 + TechTest 更新 + 文档状态同步
5. 测试    -> 全量回归（D1-D4 全部通过）-> TechTest 回填
6. 审查    -> 委托 code-reviewer 审查 D4 改动
```
